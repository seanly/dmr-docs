# AGENTS.md — issues/ 目录文档规范

本文件定义 `issues/` 目录下 issue 文档的格式规范。所有新文档必须遵循此格式。

## 文件命名

```
{编号}-{英文短横线分隔的主题}.md
```

- 编号为两位数字，从 `00` 开始递增（查看目录中已有最大编号 +1）
- 主题使用英文小写，单词间用 `-` 连接
- 示例：`00-replay-testing.md`、`28-sandbox-library-integration.md`

## 标题格式

```markdown
# {优先级}: {中文标题}（{English Title}）
```

- 优先级为 `P0`（最高）、`P1`（高）、`P2`（中）、`P3`（低）
- 如有副标题用 ` — ` 连接
- 示例：`# P1: 工具执行无超时 + 无取消（Tool Timeout and Cancellation）`

## 灵感来源引用（可选）

紧跟标题后，用 `>` 引用块说明灵感来源或背景摘要：

```markdown
> 灵感来源：TapeAgents `ReplayLLM` + `replay_tape()`
>
> 补充说明（如有）。
```

## 标准章节结构

按以下顺序组织内容，可省略不适用的章节，但不要自创章节名：

| 顺序 | 章节名 | 说明 | 必需 |
|------|--------|------|------|
| 1 | `## 问题` | 描述 DMR 当前存在的问题或不足 | 是 |
| 2 | `## TapeAgents 的做法` | 对比框架的解决方式（仅借鉴类文档需要） | 否 |
| 3 | `## DMR 的实现方案` | 具体的设计和代码方案 | 是 |
| 4 | `## 实现步骤` | 分阶段的实施清单，用 `- [ ]` 列出 | 是 |
| 5 | `## 解决的问题` | 实现后带来的价值 | 是 |
| 6 | `## 代价与风险` | 风险评估表格或列表 | 是 |
| 7 | `## 参考` | 相关代码路径、外部链接 | 否 |

## 格式要求

- **不要**使用 `---` 水平分隔线
- **不要**在章节中嵌套独立的"目标"、"背景"、"结论"等自定义章节
- 代码块标注语言类型（`go`、`python`、`bash`、`toml`、`rego` 等）
- 表格用 Markdown 标准格式
- 检查清单用 `- [ ]` 格式，放在"实现步骤"章节中

## GitHub Issue 同步

`issues/` 目录中的文档会同步创建为 `seanly/dmr` 仓库的 GitHub Issues：

- 标题：取 `# ` 后的完整标题
- 正文：标题以下的全部内容
- Labels：根据优先级打 `P0`/`P1`/`P2`/`P3`，根据分类打 `借鉴能力`/`设计缺陷`/`未来方向`
