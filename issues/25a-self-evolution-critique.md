# 自主进化架构：效率、缺陷与硬件要求的批判性分析

> 这是对 [Issue #25 自主进化架构](./25-self-evolution-architecture.md) 的冷静审视。

## 一句话总结

**Tier 0 很值得做，Tier 1 有实质性风险，微调管线对个人用户 ROI 存疑。**

---

## Tier 0（确定性缓存）— 值得做，但收益上限低于直觉

### 效率分析

| 指标 | 数值 | 说明 |
|------|------|------|
| 查找延迟 | <1ms | SQLite 单行主键查找，极快 |
| 内存占用 | ~10MB/万条 | SQLite in-file，无额外内存 |
| 命中条件 | **完全相同的问题** | MD5 hash 精确匹配 |

### 核心缺陷

**缺陷 1：个人助手场景下精确命中率极低**

精确缓存适合**重复性工作**（客服、报表、监控）。但个人全能助手的使用模式是：

```
"今天天气怎么样"        ← 每天不同答案，不可缓存
"帮我写个邮件给 XX"     ← 每次内容不同
"这段代码有什么问题"     ← 每次代码不同
"北京飞上海的航班"       ← 实时信息，缓存即错
```

真正能精确命中的场景很窄：
- 重复的系统管理命令（"查看磁盘空间"→ 总是调 `df -h`）
- 固定格式查询（"1美元等于多少人民币"→ 总是调汇率 API）

**估算**：个人助手场景下，Tier 0 精确缓存命中率可能只有 **5-15%**。

**缺陷 2：模板匹配是个伪命题**

方案中提到"从 3+ 个相似请求提取模板"。实际问题：
- **模板提取需要 NLP**——"北京时间几点"和"东京时间几点"的模板提取，本身就需要 LLM 或至少一个 NER 模型
- **参数绑定是硬问题**——知道"北京"是参数很容易，知道"北京"应该映射到工具参数 `timezone=Asia/Shanghai` 需要推理
- 纯正则/字符串模板只能覆盖极简场景，复杂一点就失效

**缺陷 3：自修正的纠正检测不可靠**

```go
// "不对" "错了" 等关键词检测
func looksLikeCorrection(text string) bool { ... }
```

问题：
- 假阳性："帮我查一下'那个不对的订单'" → 误判为纠正
- 假阴性：用户直接重新问一遍（不说"错了"，只是换个问法）→ 漏检
- 沉默失败：用户看到错误结果但没纠正，直接放弃 → 永远检测不到

### 建议

Tier 0 **只做精确缓存**，不做模板匹配。精确缓存实现简单、零风险、确实能省一些重复调用。但不要对命中率抱过高期望。

## Tier 1（本地模型）— 最大的风险点

### 硬件要求

| 模型 | 量化 | 内存需求 | Apple Silicon 推理速度 | 首 token 延迟 |
|------|------|----------|----------------------|--------------|
| Qwen2.5 7B | Q4_K_M | ~5GB | ~30-40 tok/s (M2/M3) | ~0.5-1s |
| Qwen2.5 14B | Q4_K_M | ~9GB | ~15-25 tok/s (M2/M3) | ~1-2s |
| Qwen2.5 14B | Q4_K_M | ~9GB | ~25-35 tok/s (M3 Pro/Max) | ~0.5-1s |
| Llama 3.1 8B | Q4_K_M | ~5.5GB | ~30-40 tok/s (M2/M3) | ~0.5-1s |

> 注：以上为 ollama 在 Apple Silicon 上的典型经验值。实际速度取决于具体芯片、统一内存带宽、和并发负载。

### 核心缺陷

**缺陷 1：7B-14B 模型的能力天花板很低**

TapeAgents 的 GSM8K 蒸馏实验给出了**真实数据**（`examples/gsm8k_tuning/README.md`）：

| 模型 | GSM8K 准确率 |
|------|-------------|
| Llama 3.1 8B（未微调） | **0.662**（66.2%） |
| Llama 3.1 8B（微调后） | **0.785**（78.5%） |
| Llama 3.1 70B（教师） | **0.931**（93.1%） |

关键发现：
- 微调后的 8B 仍比 70B **低 15 个百分点**
- 这还是在**单一领域**（数学题）上的微调结果
- 个人助手场景是**全领域**的，8B 模型的泛化能力远不够

**意味着**：Tier 1 路由的请求中，大约 **15-30% 会产生质量明显低于 Tier 2 的响应**。用户会频繁感到"DMR 变笨了"。

**缺陷 2：语义路由的置信度不可靠**

方案用嵌入余弦相似度决定是否路由到 Tier 1：

```
输入 → nomic-embed-text 嵌入 → 与历史 tape 余弦相似度 → >= 0.70 → Tier 1
```

问题：
- **语义相似 ≠ 答案相似**："帮我写一封感谢邮件" 和 "帮我写一封投诉邮件" 嵌入相似度很高，但需要完全不同的处理
- **0.70 阈值太低**——在 768 维嵌入空间中，0.70 的余弦相似度覆盖面非常广，会把大量"看起来像但其实不同"的请求路由到 Tier 1
- **没有回退验证**——Tier 1 返回结果后，没有机制判断结果是否正确

**缺陷 3：嵌入计算本身有成本**

```
每次请求 → 调 ollama embed API → nomic-embed-text 推理 → ~50-100ms
→ 遍历所有历史嵌入做余弦相似度 → O(n) 线性扫描
```

当历史嵌入达到 10K+ 条时，每次路由决策本身就需要 100-200ms。需要额外的 ANN 索引（如 HNSW），增加了复杂度。

**缺陷 4：ollama 常驻内存占用**

ollama 需要常驻后台才能提供 Tier 1 服务：

| 状态 | 内存占用 |
|------|----------|
| ollama 空闲（无模型加载） | ~50MB |
| 加载 7B Q4 模型 | ~5GB |
| 加载 14B Q4 模型 | ~9GB |
| 加载 14B + embed 模型 | ~10-11GB |

对于 16GB Mac 用户，常驻 10GB 给 ollama 意味着留给其他应用的内存很紧张。32GB 用户更从容。

### 建议

Tier 1 **只用于特定已验证的场景**，不做通用路由：
- 对**每种任务类型**单独评估本地模型的质量，只在质量达标的任务类型上启用
- 提高路由阈值到 0.90+，宁可漏拦（走 Tier 2）也不误拦
- 增加**结果校验**：Tier 1 响应后，用轻量规则检查是否合理

## 微调管线 — 对个人用户 ROI 最低

### 硬件要求

| 操作 | 硬件需求 | 时间 |
|------|----------|------|
| LoRA SFT（7B） | 16GB+ 统一内存 | 数小时（Apple Silicon 上用 mlx-lm） |
| LoRA SFT（14B） | 32GB+ 统一内存 | 更长 |
| 全参数 SFT | NVIDIA GPU (24GB+ VRAM) | 不适用于 Mac |
| GRPO RL | 多 GPU + vLLM | **完全不适用于个人用户** |

### 核心缺陷

**缺陷 1：数据量不足**

微调需要**数百到数千**条高质量训练样本。TapeAgents 的 GSM8K 例子用了 **3000 条**训练样本（`training_samples_3k.jsonl`）。

个人用户每天可能产生 5-20 条交互。即使全部可用（实际只有成功的才行），需要 **数月** 才能积累足够数据。而且这些数据是**全领域**的（不像 GSM8K 全是数学题），稀释了每个领域的样本密度。

**缺陷 2：微调质量退化风险**

全领域数据微调小模型的常见问题：
- **灾难性遗忘**：在 A 领域微调后，B 领域能力下降
- **过拟合用户习惯**：学到了用户的特定表达方式，但在新表达上失败
- **评估困难**：没有标准测试集，不知道微调是变好了还是变差了

**缺陷 3：工作流太重**

```
积累数据 → 导出 JSONL → 安装 mlx-lm/unsloth → 选超参 → 微调 → 合并权重 → 导入 ollama → 测试 → 发现质量不行 → 调参重来
```

这不是"自动进化"，这是机器学习工程师的日常工作。对个人用户来说门槛太高。

### 建议

**去掉微调管线**，至少在 v1 中。用 Tier 0 精确缓存 + Tier 1 few-shot 注入（从历史 tape 中检索相似交互作为示例，不微调模型本身）已经能覆盖大部分收益。

## 整体架构缺陷

### 缺陷 1：进化方向不可控

系统自动学习意味着用户**不知道**它学了什么、会在什么时候触发。某天突然一个请求被缓存拦截返回了过时信息，用户会困惑"DMR 怎么变傻了"。

**需要**：每次缓存命中时明确告知用户（例如 `[cached response]` 标记），让用户可以选择"这次不用缓存，走 LLM"。

### 缺陷 2：存储增长

| 组件 | 6个月估算 |
|------|-----------|
| patterns.db（精确缓存） | ~10-50MB |
| 嵌入向量（768维 × 10K） | ~30MB |
| 训练 JSONL | ~50-200MB |
| 合计 | ~100-300MB |

不算多，但需要定期清理策略。

### 缺陷 3：InterceptInput 的原子性问题

当前 `InterceptInput` hook 只能返回纯文本（`InterceptResult{Output, Kind}`）。但很多 agent 交互涉及**工具调用链**——Tier 0 缓存的"调用 df -h 然后解析结果"需要实际执行工具，不能只返回文本。

这意味着 Tier 0 能缓存的场景进一步收窄到**纯文本问答**，不包括工具调用场景。

要支持工具调用缓存，需要修改核心代码（让 InterceptResult 支持工具调用序列），违背了"零侵入"的设计目标。

## 硬件需求总结

### 最低配置（只跑 Tier 0）

| 组件 | 要求 |
|------|------|
| CPU | 任意 |
| 内存 | 不额外占用（SQLite 文件存储） |
| 磁盘 | +50MB |
| GPU | 不需要 |

### 推荐配置（Tier 0 + Tier 1）

| 组件 | 要求 |
|------|------|
| 芯片 | Apple M2/M3 或更高 |
| 统一内存 | **32GB**（16GB 太紧张——7B 模型 5GB + embed 模型 1GB + macOS + 其他应用） |
| 磁盘 | +15GB（模型文件 ~10GB + 数据 ~5GB） |
| 软件 | ollama |

### 微调配置（不推荐个人用户）

| 组件 | 要求 |
|------|------|
| 芯片 | Apple M2 Pro/Max 或更高 |
| 统一内存 | **64GB**（微调 7B LoRA 需要 ~20-30GB，14B 需要更多） |
| 磁盘 | +30GB |
| 软件 | ollama + mlx-lm 或 Python ML 工具链 |
| 替代 | 租用 GPU 云服务（RunPod、Lambda Labs） |

## 修正后的建议

| Phase | 内容 | ROI | 复杂度 |
|-------|------|-----|--------|
| **Phase 1** | Tier 0 精确缓存（只做 exact match） | 中（省 5-15% LLM 调用） | 低（~300 行 Go） |
| **Phase 2** | 缓存命中标记 + 用户跳过机制 | 高（信任/可控性） | 低 |
| **Phase 3** | Tier 1 **受限路由**（只对验证过的任务类型） | 中 | 中 |
| ~~Phase 4~~ | ~~微调管线~~ | ~~低~~ | ~~高~~ |

**最现实的短期收益**：Tier 0 精确缓存 + cron 任务模式复用。Cron 是最完美的场景——重复性高、模式固定、不需要实时信息。

## 补充数据：GAIA 基准测试（复杂 Agent 任务）

TapeAgents 的 GAIA Agent 基准测试（`examples/gaia_agent/README.md`）提供了**复杂 agent 任务**中不同模型的对比数据——这比单纯的数学题（GSM8K）更接近 DMR 的实际场景：

| 模型 | 验证集准确率 | 测试集准确率 | 说明 |
|------|------------|------------|------|
| Sonnet 3.7 maj@3 | **55.8%** | — | 3 次投票 |
| Sonnet 3.7 | **53.9%** | — | 当前最佳 |
| GPT-4o | **37.0%** | **33.2%** | 大模型 |
| GPT-4o mini maj@3 | **32.3%** | — | 小模型 3 次投票 |
| GPT-4o mini | **27.3%** | **21.9%** | 小模型 |

### 对 Tier 1 的影响

**GPT-4o mini 的质量仅为 GPT-4o 的 66%**（测试集 21.9% vs 33.2%）。而 GPT-4o mini 的参数量远大于本地 7B-14B 模型。这意味着：

- 本地 7B 模型在复杂 agent 任务上的质量可能只有大模型的 **50% 甚至更低**
- **GAIA 任务比 GSM8K 更接近 DMR 场景**（多步推理、工具调用、信息检索），所以 GAIA 的数据更有参考价值
- GSM8K 的 84.3%（8B vs 70B）可能给出了**过于乐观**的估计

### 集成投票是一个亮点

GPT-4o mini 通过 **3 次投票**（maj@3）从 27.3% 提升到 32.3%（+5 百分点），接近 GPT-4o 单次（37.0%）的 87%。

这暗示：如果确实要用 Tier 1，**多次推理 + 投票** 可能比单次推理更有效。代价是 3 倍推理时间，但本地推理边际成本为零。这可以纳入 Phase 3 的设计。

### 其他补充数据

- **GRPO RL（1B 模型）**：Llama 3.2 1B 在 GSM8K 上从 44.4% → ~53%，但需要 **H100 GPU 训练 2 小时**，完全不适用于个人用户
- **缓存命中率**：TapeAgents 代码中 **没有任何缓存命中率统计**（`CachedLLM` 只有逐条 log，无聚合计数），说明即使框架作者也没有收集这方面数据
- **HotPotQA prompt 优化**：结果"非常不稳定"（README 原文），小数据集上的优化高度依赖随机性
